Appendix B: The MVP Outline (PWA)
This build plan is forged from a ruthless, first-principles pivot. Our MVP R&D (the 4-day "TMA" build) revealed a fatal flaw in that strategy: a high-friction adoption path for a "zero-follower" product.
This new PWA-first plan is the correct, validated path. It is architected for minimum friction and maximum data collection, solving the "Cold Start" problem.
1. Governing Principles (The "No Waste" Mandate)
Philosophical Foundation (The "Why"): The engine is named AIRE (Ascending Infinite Recursion Engine) because it is the tangible tool to build a world led by WayMakers, "by giving them AIRE to breathe." It is an act of Kairos, the opportune, decisive moment, for a Participant to consciously intervene in their own life's loop and choose ascent over devolution. Every feature must serve this mission.
All Signal, No Noise: Every feature must directly serve the core PICV (Prime > Improve > Commit > Visualize) loop.
Forge the Engine First: We are building the core engine. All "Strive Operator" (Phase 2) features are deferred.
Solve for "Day 2": The single biggest risk is participant churn. The re-engagement loop is not a feature; it is the core of the product.
PWA-First: We will build a lightweight, "online-only" Progressive Web App. This is a conscious trade-off: we sacrifice offline capability for a 10x reduction in build time and complexity.
2. The Core Tech Stack
Deployment: Vercel (for instant, global CDN deployment).
Backend & Auth: Supabase (for its simple Auth and powerful Postgres database, which ensures our migration path is frictionless).
Domain: striveos.io
Code: "Vibe coded" with AI assistance (e.g., Cursor) using standard web technologies (HTML/JS/CSS, likely with a framework like React or Svelte, however the best option will be utilized).
3. The Core User Experience: The PICV Daily Cycle
The entire MVP is a "Dashboard-first" experience. The Participant opens the app to their main dashboard, which is the "home base" that provides context, visual reward, interaction with their own ascent (via Kairos), and the gateway to the daily ritual.
Core Technical Requirement: Keystroke Persistence: The app must auto-save text input in real-time (e.g., to browser localStorage). Losing a Participant's Prime or Improve text due to a page refresh or crash is a fatal, retention-killing flaw and will be prevented.
Screen 0: The Dashboard (The "Home Base")
Function: This is the first screen a Participant sees upon opening the app. It provides immediate context and value.
UI/UX:
"Welcome to your Daily AIRE"
The Ascent Graph: The line chart displaying the Participant's "Execution Score" (1-10) over time. This is the primary visual motivator.
The "Begin Cycle" Button: A single, clear call-to-action that initiates the PICV loop.
The "Improvement Log" (Link): A link or tab to view the log of all past Improve text entries.
Kairos (Post-MVP): A chat icon to interact with the "AI Mirror" (defined in Section 6).
Screen 1: PRIME
Function: To prime the Participant's state for constructive reflection. Initiated from the Dashboard.
Prompt: "What is one experience, gift, or opportunity you are grateful for right now, and why?"
Action: User inputs prime text and proceeds.
Screen 2: IMPROVE
Function: The logical hub. It fetches the Previous Commitment and facilitates reflection and rating.
UI/UX (Logic Branch A: Cycle 2+ User):
Displays the Previous Commitment text.
Part 1: Rate: "On a scale of 1â€“10, how well did you execute this commitment?" (Slider or 10-star row).
Part 2: Reflect (Learn): "What is the most powerful lesson learned from executing this commitment?" (Text input).
UI/UX (Logic Branch B: First-Time User):
Part 1: Reflect (Learn): "What is the most valuable thing you learned yesterday?" (Text input).
Action: User inputs score (if applicable) and improve text, then proceeds.
Screen 3: COMMIT
Function: To set a clear, high-leverage intent for the next cycle.
Prompt: "What is the one decisive action that, when you execute it today, will move you unmistakably forward?"
Action: User inputs commit text. This commit text becomes the Previous Commitment for the next cycle.
Screen 4: VISUALIZE (Summary & Save Screen)
Function: A final confirmation and data-save trigger. This is the end of the daily loop.
UI/UX: Displays all data points from the current cycle (Prime text, Score, Improve text, Commit text) for final review.
Action: A single button: "Save & Forge Forward."
Tech: Clicking this button saves the complete cycle (all 4 data points) as a single record to the Supabase backend. The app then routes the Participant back to the Dashboard (Screen 0), where they will see their newly updated "Ascent Graph."
5. Critical Path Solutions (The "Anti-Failure" Mechanics)
Solving the "Day 2 Re-engagement Problem" (Email/SMS Loop):
Onboarding: After the first PICV cycle, the app must ask for the Participant's email and/or phone number and their preferred time for their next cycle.
The Engine: A simple, automated cron job (e.g., Vercel Cron or Supabase scheduled function) will send a daily, single-link notification at their chosen time.
The Notification: "Your next AIRE cycle is ready. Begin your ascent." (This link deep-links them directly into the PWA). This is our primary, reliable re-engagement channel.
Contingency: If SMS notification is plausible with similar complexity (e.g., via Twilio), it will be offered as the primary option over email.
Solving the "Usability Problem" (Online-Only):
We will consciously de-scope offline capability for the MVP.
The app will have a simple "offline detector" that displays a clear message if no connection is present. We temporarily accept this trade-off for speed.
Solving the "Scalability Problem" (Migration Path):
Validation Metric: We will define our internal "Go" signal: (e.g., "Achieve 30% Day-7 retention from our first 100 active participants"). This is a business metric to trigger funding for Phase 2, not a technical failure point. The PWA will function indefinitely.
The Path: Because all user data (Auth, PICV entries) is in our portable Supabase backend, there is zero migration friction. When we hit our metric and fund Phase 2/3, the new native app will connect to this exact same backend. Participants will log in and see their entire history instantly. The PWA is a disposable front-end; the Supabase backend is the permanent asset.
6. Post-MVP Evolution: Advanced Features
The lean MVP is designed to validate the core PICV loop. The data and revenue generated will directly fund the development of the true features that make AIRE a "Personal Protocol Engine." These are the immediate next steps:
The "Resonance Engine" (Contextual Improvement):
Function: This feature will transform the Improvement Log from a passive archive into a dynamic, living asset.
Implementation: During the COMMIT phase, the system will intelligently query the Participant's Improvement Log (based on keywords or, later, AI-driven semantic analysis) and surface the most relevant past improvements to help inform their new commitment.
The "AI Mirror" (Socratic Guidance):
Function: An integrated AI (our "Kairos") that acts as a ruthless strategic mirror, not a passive assistant.
Implementation: The AI will be permissioned to read a Participant's entire data set (Ascent Graph, Prime, Improve, Commit, and Score history). It can then provide holistic, Socratic feedback, e.g., "I notice your execution scores are 30% higher on days your Prime gratitude is family-focused. What does this tell you?" or "Your Commitment today is very similar to one from three weeks ago where you scored a 4/10. What Improvement will you deploy to ensure a different outcome?"
Note: This feature will be available both at the dashboard, and as an assistant through out each AIRE cycle to help maximize progress.
